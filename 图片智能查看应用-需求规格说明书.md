# 图片智能查看应用 — 需求规格说明书

> **版本**: v2.0
> **技术栈**: Electron + React + TypeScript
> **目标平台**: Windows
> **文档日期**: 2026-04-13

---

## 1. 项目概述

### 1.1 项目背景

当前市场上缺乏一个集**图片元数据分析**、**AI 内容识别**、**插件化布局**于一体的图片管理工具。用户需要一个能够智能整理归类图片、灵活展示的创意应用。

### 1.2 项目目标

构建一款面向 Windows 平台的桌面图片智能查看应用，核心能力包括：

- 深度解析图片元数据（EXIF / IPTC / XMP）
- AI 驱动的语义内容识别与智能分类
- 可插拔的插件系统，支持自定义布局与功能扩展
- 本地化数据持久化，支持离线使用

### 1.3 目标用户

- 摄影爱好者：需要查看和管理大量照片的元数据信息
- 设计师：需要灵活的图片浏览和布局方式
- 普通用户：需要智能分类整理个人图片库

## 3. 功能需求

### 3.1 图片库管理

> **优先级**: P0（核心功能）

图片库是应用的基础数据管理模块，负责图片的导入、组织和检索。

#### 3.1.1 图片导入

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户选择导入**文件夹**（递归扫描文件夹及所有子文件夹中的图片）或**单个图片文件** |
| **处理逻辑** | ① 扫描目标路径下的所有图片文件（文件夹模式递归遍历子文件夹） ② 逐张计算文件 MD5 哈希值 ③ 检查数据库中是否已存在相同 MD5 ④ 若已存在：仅更新 `file_path` 为最新真实路径，跳过后续处理 ⑤ 若不存在：创建新记录入库 ⑥ 异步触发元数据解析和 AI 识别，完成后更新数据库信息 ⑦ 元数据和 AI 识别均处理完毕后，标记该图片为「处理完成」 |
| **预期结果** | 新图片入库并异步完成全部分析；重复图片仅更新路径，不重复处理 |
| **扫描深度** | 文件夹模式无层级限制，递归扫描所有层级的子文件夹 |

#### 3.1.2 重复图片检测

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户导入图片时，系统检测到数据库中已存在相同 MD5 值 |
| **处理逻辑** | ① 将图片主表的 `file_path` 更新为当前最新真实路径 ② 跳过元数据解析和 AI 识别，直接复用已有分析数据 ③ 进度标记为 `skipped` |
| **预期结果** | 不产生重复记录，不重复消耗解析和 AI 资源，已有元数据和识别结果自动关联到最新路径 |
| **典型场景** | 同一图片被移动到不同文件夹后再次导入；同一图片在不同目录中存在副本 |

#### 3.1.3 智能搜索

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户输入搜索关键词或选择筛选条件 |
| **处理逻辑** | 根据文件名、元数据字段、AI 标签、分类、**图片内文字（全文索引）**等多维度匹配 |
| **全文索引** | AI 识别的文字内容（`text.content`）存入 SQLite FTS5 全文索引表，支持中文分词和模糊匹配 |
| **预期结果** | 实时展示匹配的图片列表，搜索结果中高亮匹配的文字片段 |

#### 3.1.4 分类浏览

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户选择某个分类标签（如风景、人像、美食等） |
| **处理逻辑** | 从数据库检索该分类下的所有图片 |
| **预期结果** | 展示该分类下的图片集合 |

提供图片元数据深度解析能力，提取并展示图片的详细技术信息。

#### 3.2.1 EXIF 数据解析

| **处理逻辑** | 解析 EXIF 数据段，提取相机型号、镜头、光圈、快门速度、ISO、GPS 坐标、拍摄时间等 |
| **预期结果** | 在信息面板中结构化展示 EXIF 数据 |

#### 3.2.2 图片基本信息展示

| 项目 | 说明 |
|------|------|
| **触发条件** | 用户查看任意图片 |
| **处理逻辑** | 读取图片文件头信息，获取尺寸、格式、色彩空间、位深度、文件大小等 |
| **预期结果** | 展示图片的基本属性信息 |

#### 3.2.3 支持的元数据格式

| 格式 | 说明 |
|------|------|

### 3.3 AI 语义内容识别

> **优先级**: P1（重要功能）

集成 AI 图像识别服务，自动分析图片的语义内容，实现智能分类和标签管理。采用**四维度一次识别**策略，单次 API 调用同时返回物体、场景、活动、情绪四个维度的分析结果。文字识别（OCR）作为独立模块，支持用户在设置中选择识别方案（AI / PaddleOCR / Tesseract）。

#### 3.3.1 识别维度概览

| 维度 | 说明 | 示例 |
|------|------|------|
| **物体识别** | 识别图片中的具体物体，标注边界框 | 人、猫、狗、车、建筑、家具 |
| **场景识别** | 识别图片所处的环境场景类型 | 海滩、办公室、卧室、街道、山林、餐厅 |
| **活动识别** | 识别图片中人物的动作或事件 | 跑步、吃饭、开会、旅行、购物、运动 |
| **情绪识别** | 识别图片表达的情感氛围 | 欢乐、宁静、悲伤、兴奋、温馨、紧张 |

> **设计决策**：四个维度在一次 prompt 中完成。文字识别（OCR）独立为可配置模块（详见 3.10），原因：① OCR 有成熟的本地方案（PaddleOCR、Tesseract），无需依赖 AI API ② 本地 OCR 无网络延迟和 API 成本 ③ 用户可根据精度/速度/离线需求自由选择方案 ④ 本地 OCR 与人物识别共用队列串行执行，减少 Worker Thread 资源占用。

#### 3.3.2 AI 服务模式

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| 云端 API | 用户选择云端模式且网络可用 | 调用云端 API（如 GPT-4o Vision、AWS Rekognition） |
| 本地模型 | 用户选择本地模式或无网络 | 使用本地部署的 AI 模型进行推理（如 Qwen2.5-VL） |
| 模式切换 | 用户在设置中切换 AI 模式 | 立即应用新模式，后续识别使用新配置 |

#### 3.3.3 识别类型标识

每张图片完成 AI 识别后，系统必须标识该图片**包含哪些识别类型**，用于 UI 展示和搜索筛选。

| 标识字段 | 类型 | 说明 |
|---------|------|------|
| `detectedTypes` | string[] | 该图片实际识别出的维度列表，取值：`"object"` / `"scene"` / `"activity"` / `"emotion"` / `"text"` |

> **注意**：`"text"` 维度由文字识别模块（3.10）独立写入，不属于 AI 识别的输出。

**示例**：
- 一张海滩跑步照片 → `["object", "scene", "activity"]`
- 一张纯风景照（无人） → `["object", "scene"]`
- 一张温馨家庭合照 → `["object", "scene", "activity", "emotion"]`
- 一张菜单照片 → `["object", "scene", "text"]`（text 由 OCR 模块写入）
- 一张路牌照片 → `["object", "scene", "text"]`（text 由 OCR 模块写入）

#### 3.3.4 智能分类与标签

| 项目 | 说明 |
|------|------|
| **自动分类** | 根据识别结果自动归类到预设类别（风景/人像/美食/建筑/运动/家庭等） |
| **标签生成** | 自动生成描述性标签，支持用户手动添加、编辑、删除 |
| **置信度** | 每个识别结果附带置信度分数（0-1），辅助用户判断准确性 |
| **类型标识** | 每张图片标注已识别的维度类型，便于 UI 状态展示和筛选 |

#### 3.3.5 AI 识别返回数据格式

> **重要说明**：`md5` 和 `timestamp` 字段由**本地应用**在收到 AI 响应后自动附加，**不属于 AI 模型的输出内容**。AI 提示词不应要求模型返回这两个字段。以下 JSON 展示的是**最终存储到数据库的完整结构**（含本地附加字段）。
>
> **为什么 MD5 基于原图而非 AI 输入？** AI 识别使用的是经过预处理压缩的图片（详见 3.3.7），而非原始图片文件。MD5 必须基于原始图片文件计算，以确保文件唯一标识的准确性和一致性——同一张原图无论经过何种压缩参数处理，其 MD5 始终不变。

**统一格式**（四维度：物体 + 场景 + 活动 + 情绪，单次调用返回）：

```json
{
  "md5": "图片文件MD5哈希值（本地计算，非AI生成）",
  "timestamp": "识别完成时间戳 ISO 8601（本地生成，非AI生成）",
  "detectedTypes": ["object", "scene", "activity", "emotion"],
  "objects": [
    {
      "name": "物体名称",
      "confidence": 0.95,
      "boundingBox": {
        "x": 100,
        "y": 50,
        "width": 200,
        "height": 150
      }
    }
  ],
  "scene": {
    "type": "场景类型",
    "confidence": 0.88
  },
  "activity": {
    "type": "活动类型",
    "confidence": 0.82
  },
  "emotion": {
    "type": "情绪类型",
    "confidence": 0.90
  },
  "category": "自动分类（风景/人像/美食/建筑/运动/家庭等）",
  "tags": ["标签1", "标签2", "标签3"]
}
```

> **文字识别数据**：当用户选择「AI」方案时，AI 返回的 `text` 字段会额外附加到上述结构中（格式同 3.10.3）；当用户选择本地 OCR 方案时，`text` 字段由 OCR 模块独立写入。两种方案的输出格式统一，均包含 `text.content` 和 `text.regions`。

**字段说明：**

| 字段 | 类型 | 必填 | 数据来源 | 说明 |
|------|------|:----:|---------|------|
| `md5` | string | ✅ | 本地计算 | 图片文件 MD5 哈希，由本地应用对原始图片文件计算后附加，唯一标识 |
| `timestamp` | string | ✅ | 本地生成 | 识别完成时间，ISO 8601 格式，由本地应用在 AI 响应返回后附加 |
| `detectedTypes` | string[] | ✅ | AI 输出 | 实际识别出的维度列表：`"object"` / `"scene"` / `"activity"` / `"emotion"` / `"text"` |
| `objects` | array | ✅ | AI 输出 | 物体识别结果数组，可为空数组 `[]` |
| `objects[].name` | string | ✅ | AI 输出 | 物体名称 |
| `objects[].confidence` | number | ✅ | AI 输出 | 置信度，范围 0–1 |
| `objects[].boundingBox` | object | ❌ | AI 输出 | 物体边界框坐标（x, y, width, height） |
| `scene` | object | ✅ | AI 输出 | 场景识别结果 |
| `scene.type` | string | ✅ | AI 输出 | 场景类型 |
| `scene.confidence` | number | ✅ | AI 输出 | 置信度，范围 0–1 |
| `activity` | object | ❌ | AI 输出 | 活动识别结果，无活动时为 `null` |
| `activity.type` | string | ✅ | AI 输出 | 活动类型 |
| `activity.confidence` | number | ✅ | AI 输出 | 置信度，范围 0–1 |
| `emotion` | object | ❌ | AI 输出 | 情绪识别结果，无明显情绪时为 `null` |
| `emotion.type` | string | ✅ | AI 输出 | 情绪类型 |
| `emotion.confidence` | number | ✅ | AI 输出 | 置信度，范围 0–1 |
| `category` | string | ✅ | AI 输出 | 自动分类结果 |
| `tags` | array | ✅ | AI 输出 | 描述性标签数组 |

> **文字识别字段**（`text`、`text.content`、`text.regions`）由文字识别模块（3.10）独立写入，不属于 AI 识别输出。字段格式详见 3.10.3。

#### 3.3.6 AI 提示词规范

> 适用于 GPT-4o Vision、Qwen2.5-VL 等多模态大模型的提示词设计规范。

**设计原则**：

| 原则 | 说明 |
|------|------|
| 单次调用 | 四个维度在一次 prompt 中完成，减少 API 调用次数和延迟，节省 token |
| 预设分类 | 给出有限的分类选项列表，避免模型自由发散导致分类不可控 |
| 低温度 | `temperature = 0.1`，保证 JSON 格式稳定、分类结果一致 |
| 尺寸注入 | prompt 中注入图片像素尺寸，提高边界框坐标准确度 |
| 后处理校验 | 代码层对 `detectedTypes`、边界框越界、置信度范围做二次修正 |

**System Prompt**：

```
你是一个专业的图片语义分析引擎。你的任务是对用户提供的图片进行四维度语义分析：物体识别、场景识别、活动识别、情绪识别。

## 分析规则

### 1. 物体识别（object）
- 识别图片中所有可见的具体物体
- 每个物体需标注名称和置信度（0-1）
- 尽可能提供边界框坐标（boundingBox），坐标为相对于图片原始尺寸的像素值
- 边界框格式：{ "x": 左上角x, "y": 左上角y, "width": 宽度, "height": 高度 }
- 如果无法准确估计边界框，省略 boundingBox 字段

### 2. 场景识别（scene）
- 判断图片所处的环境场景类型
- 从以下预设场景中选择最匹配的一个：
  海滩、山脉、森林、公园、街道、城市天际线、办公室、卧室、客厅、厨房、
  餐厅、教室、健身房、商场、机场、车站、博物馆、图书馆、医院、
  剧院、体育场、停车场、农田、花园、湖泊、河流、沙漠、雪地、其他
- 如果图片无明显场景特征（如纯色背景、微距特写），type 填 "其他"

### 3. 活动识别（activity）
- 判断图片中是否有人物正在进行某种活动
- 从以下预设活动中选择最匹配的一个：
  跑步、走路、骑行、游泳、做饭、吃饭、喝饮料、工作（电脑）、开会、
  读书、写字、画画、演奏乐器、唱歌、跳舞、运动（球类）、健身、
  瑜伽、拍照、旅行、购物、开车、聊天、拥抱、玩耍、睡觉、其他
- 如果图片中无人或人未进行明显活动，activity 填 null

### 4. 情绪识别（emotion）
- 判断图片整体表达的情感氛围
- 从以下预设情绪中选择最匹配的一个：
  欢乐、宁静、悲伤、兴奋、温馨、紧张、孤独、震撼、神秘、浪漫、其他
- 如果图片无明显情绪特征（如产品图、文档截图），emotion 填 null

### 5. 自动分类（category）
- 根据以上四个维度的综合分析，从以下类别中选择最匹配的一个：
  风景、人像、美食、建筑、运动、家庭、动物、交通、艺术、科技、其他

### 6. 标签生成（tags）
- 生成 3-8 个描述性标签
- 标签应涵盖：主要物体、场景特征、颜色色调、氛围风格、情绪感受等
- 标签使用简短的名词或形容词短语

### 7. detectedTypes
- 根据实际分析结果动态生成
- 如果识别到物体 → 包含 "object"
- 如果识别到场景（非"其他"） → 包含 "scene"
- 如果识别到活动（非 null） → 包含 "activity"
- 如果识别到情绪（非 null） → 包含 "emotion"
- 注意：`"text"` 由文字识别模块（3.10）独立写入，不在此处生成

## 输出要求
- 必须且仅输出合法的 JSON，不要包含任何其他文字、解释或 markdown 代码块标记
- 严格按照下方 JSON 模板输出，不要增减字段
- 置信度范围 0-1，保留两位小数
- 图片尺寸为 {width}x{height} 像素，边界框坐标需基于此尺寸
```

**User Prompt 模板**（调用时由应用填充 `{width}`、`{height}`）：

```
请分析这张图片，返回 JSON 结果。

图片尺寸：{width} x {height}

JSON 模板：
{
  "detectedTypes": [],
  "objects": [
    { "name": "物体名称", "confidence": 0.00, "boundingBox": { "x": 0, "y": 0, "width": 0, "height": 0 } }
  ],
  "scene": { "type": "场景类型", "confidence": 0.00 },
  "activity": { "type": "活动类型", "confidence": 0.00 },
  "emotion": { "type": "情绪类型", "confidence": 0.00 },
  "category": "分类",
  "tags": ["标签1", "标签2", "标签3"]
}
```

> **注意**：模板中**不包含** `md5` 和 `timestamp` 字段。这两个字段由本地应用在收到 AI 响应后自动附加：`md5` 取自导入时对原始图片文件计算的哈希值，`timestamp` 取自 AI 响应返回的本地时间。

**后处理校验规则**：

模型输出可能不完全合规，代码层必须执行以下校验和修正：

| 校验项 | 规则 |
|--------|------|
| `detectedTypes` 重建 | 不信任模型自行判断，根据实际输出内容重新计算：objects 非空 → 加 `"object"`；scene.type 非"其他" → 加 `"scene"`；activity 非 null → 加 `"activity"`；emotion 非 null → 加 `"emotion"`。`"text"` 由 OCR 模块独立写入，不在此处处理 |
| `activity` 空值 | 如果 `activity.type` 为空或不存在，整字段置 `null` |
| `emotion` 空值 | 如果 `emotion.type` 为空或不存在，整字段置 `null` |
| 边界框坐标映射 | AI 返回的边界框坐标基于压缩图尺寸，必须映射回原图尺寸：`原图坐标 = AI坐标 × (原图尺寸 / 压缩图尺寸)` |
| 边界框越界 | 映射后的 `x`/`y` 钳制到 `[0, 原图Width]`/`[0, 原图Height]`，`width`/`height` 钳制到 `[0, 原图Width - x]`/`[0, 原图Height - y]` |
| 置信度范围 | 所有 `confidence` 值钳制到 `[0, 1]` |
| JSON 解析失败 | 重试最多 2 次；仍失败则记录错误日志，该图片标记为"识别失败" |

**调用参数**：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `temperature` | `0.1` | 低温度保证输出稳定 |
| `max_tokens` | `1024` | 足够容纳完整 JSON |
| `response_format` | `{ "type": "json_object" }` | OpenAI 原生 JSON 模式（如支持） |

#### 3.3.7 图片预处理压缩

> **优先级**: P0（核心功能）

AI 服务（云端 API / 本地模型）对输入图片大小有严格限制，过大的图片会导致调用失败或超时。因此，在提交给 AI 识别之前，必须对原图进行**预处理压缩**，将几 MB 的原图压缩到几百 KB 级别。

**核心原则**：压缩仅用于 AI 识别的输入，**不影响原图存储和 MD5 计算**。MD5 始终基于原始图片文件计算。

##### 3.3.7.1 压缩流程

```
原图文件（几 MB）
    │
    ▼
读取原图尺寸（width × height）
    │
    ▼
是否超过最大尺寸限制？
    │ 是                          │ 否
    ▼                             ▼
等比缩放到最大尺寸以内         跳过缩放
    │                             │
    └──────────┬──────────────────┘
               ▼
无损压缩编码（WebP lossless / PNG 优化）
    │
    ▼
检查压缩后文件大小
    │
    ├── ≤ 目标大小（如 500KB）──▶ 输出压缩图，提交 AI
    │
    └── ＞ 目标大小 ──▶ 逐步降低质量参数（切换为有损模式）
                              │
                              ▼
                         重新压缩，直到 ≤ 目标大小
```

##### 3.3.7.2 压缩参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **最大像素尺寸** | `4096 × 4096` | 超过此尺寸的图片等比缩放，保持宽高比 |
| **目标文件大小** | `≤ 500 KB` | 压缩后文件大小的上限 |
| **优先编码格式** | WebP | 同等质量下体积最小，主流 AI 服务均支持 |
| **压缩策略** | 优先无损 → 超限时切换有损 | 在保证视觉质量的前提下尽量减小体积 |
| **有损质量下限** | `quality ≥ 75` | 有损模式的最低质量阈值，避免过度失真影响识别精度 |

##### 3.3.7.3 压缩与原图的关系

| 项目 | 原图 | 压缩图（AI 识别用） |
|------|------|-------------------|
| **用途** | 本地存储、展示、元数据解析 | 仅作为 AI 识别的输入 |
| **MD5** | ✅ 基于原图计算，作为唯一标识 | ❌ 不计算 MD5 |
| **存储** | 保留原始文件，不做任何修改 | 临时生成，用后可清理（或缓存复用） |
| **尺寸** | 原始尺寸 | 最大 4096 × 4096 |
| **体积** | 原始大小（可能几 MB~几十 MB） | ≤ 500 KB |

##### 3.3.7.4 边界框坐标映射

由于压缩图可能经过缩放，AI 返回的边界框坐标基于压缩图尺寸，后处理时需要映射回原图尺寸：

```
原图坐标 = AI返回坐标 × (原图尺寸 / 压缩图尺寸)
```

> 此映射在后处理校验阶段自动完成，对上层业务透明。

##### 3.3.7.5 压缩图缓存

| 项目 | 说明 |
|------|------|
| **缓存位置** | 应用数据目录下的 `cache/compressed/` 文件夹 |
| **缓存键** | `{md5}_{maxSize}_{targetKB}.webp` |
| **缓存策略** | 基于 MD5 判断，同一张图片只压缩一次，后续直接复用 |
| **缓存清理** | 应用提供「清理缓存」功能，可手动或自动（超过 30 天未访问）清理 |

### 3.6 本地数据存储

> **优先级**: P0（核心功能）

使用 SQLite 本地数据库存储所有分析数据，确保数据持久化和快速访问。

#### 3.6.1 数据模型

**核心设计原则**：所有数据通过图片的 **MD5 哈希值** 作为唯一标识进行关联。图片主表中的 `file_path` 始终指向该图片的**最新真实路径**——当同一 MD5 的图片在不同位置被再次导入时，仅更新路径，不重复处理。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   图片主表    │     │  元数据表     │     │ AI 分析结果表 │
│  (images)    │────▶│  (metadata)  │     │  (ai_result) │
│              │     │              │     │              │
│ • md5 (PK)   │     │ • md5 (FK)   │     │ • md5 (FK)   │
│ • file_path  │     │ • exif_data  │     │ • version    │
│   (最新路径) │     │ • iptc_data  │     │ • detected   │
│ • file_name  │     │ • xmp_data   │     │   _types     │
│ • file_size  │     │ • basic_info │     │ • objects    │
│ • format     │     └──────────────┘     │ • scene      │
│ • width      │                          │ • activity   │
│ • height     │     ┌──────────────┐     │ • emotion    │
│ • import_time│     │  插件配置表   │     │ • emotion    │
└──────────────┘     │  (plugins)   │     │ • category   │
                     └──────────────┘     │ • tags       │
                                          │ • text       │
                                          │   (content  │
                                          │    +regions) │
                                          └──────────────┘
```

#### 3.6.2 存储场景

| 场景 | 行为 |
|------|------|
| 图片导入（新图片） | 计算 MD5 → 创建图片主表记录 → 异步触发元数据解析和 AI 识别 |
| 图片导入（MD5 重复） | 检测到已存在相同 MD5 → 仅更新 `file_path` 为最新真实路径 → 跳过元数据解析和 AI 识别 → 进度标记 `skipped` |
| 元数据解析完成 | 通过 MD5 关联，将 EXIF/IPTC/XMP 数据写入元数据表 |
| AI 识别完成 | 通过 MD5 关联，将识别结果写入 AI 分析结果表，文字内容同步写入 FTS5 全文索引表 |
| 图片文件变化 | 检测文件变更，更新数据库记录，保持数据一致性 |
| 用户修改标签 | 更新 AI 分析结果表中的 tags 字段 |
| 搜索/筛选 | 从本地数据库多条件查询，返回匹配结果 |

#### 3.6.3 离线支持

- 所有已分析数据存储在本地 SQLite 数据库中
- 无网络环境下可正常浏览、搜索、查看已分析图片的元数据和 AI 结果
- 仅 AI 云端识别功能需要网络连接

### 3.7 处理进度追踪

> **优先级**: P0（核心功能）

图片导入后会触发三条异步处理线：本地处理（MD5 计算 + 元数据解析）、本地识别（人物识别 → 文字识别，串行）、AI 识别。用户需要一个统一的入口实时查看各类处理的进度状态。

#### 3.7.1 进度状态定义

每张图片在导入后，会经历以下处理阶段：

| 阶段 | 处理方 | 说明 |
|------|--------|------|
| **MD5 计算** | 本地 | 对原始图片文件计算 MD5 哈希值，用于去重判断和唯一标识（本地处理的第一步） |
| **元数据解析** | 本地 | 解析 EXIF / IPTC / XMP 等元数据信息 |
| **人物识别** | 本地 | InsightFace ONNX 人脸检测、特征提取、person_id 匹配 |
| **文字识别** | 本地/AI | 根据配置方案执行 OCR（PaddleOCR / Tesseract / AI），与人物识别串行 |
| **AI 识别** | AI 服务 | 调用云端 API 或本地模型进行四维度语义内容识别（物体/场景/活动/情绪） |

> **处理完成条件**：一张图片的「本地处理」（MD5 + 元数据）、「本地识别」（人物识别 + 文字识别）和「AI 识别」均完成后，才标记为「处理完成」。重复图片（MD5 已存在）直接标记为 `skipped`，视为已处理。
>
> **串行说明**：人物识别和文字识别共用一个 Worker Thread，同一张图片先执行人物识别，完成后再执行文字识别。选择 AI 方案时，文字识别结果由 AI 识别队列返回，不占用本地队列。

每张图片的各阶段状态：

| 状态 | 说明 |
|------|------|
| `pending` | 等待处理（排队中） |
| `processing` | 正在处理 |
| `completed` | 处理完成 |
| `failed` | 处理失败（附带错误信息） |
| `skipped` | 已跳过（如重复图片复用已有数据） |

#### 3.7.2 进度面板 UI

| 项目 | 说明 |
|------|------|
| **入口位置** | 应用主界面底部状态栏 + 可展开的进度面板（点击状态栏进度指示器展开） |
| **总进度** | 底部状态栏显示当前批次的总进度：`已完成数 / 总数`，以及百分比进度条 |
| **独立状态** | 三条处理线（本地处理、人物识别、AI 识别）各自独立运行，各有独立状态，但不各自显示进度条 |
| **图片级状态** | 展开面板中，每张图片为一个条目，同时显示三条线的独立状态图标 |
| **实时更新** | 每张图片任一状态变化时实时刷新，无需手动刷新 |
| **错误提示** | 失败的图片以醒目方式标记，点击可查看错误详情 |
| **操作按钮** | 提供「重试失败项」「暂停」「取消全部」等操作 |

| `ocr_error` | string | 文字识别失败时的错误信息，成功时为 `null` |
| `ai_status` | string | AI 识别状态：`pending` / `processing` / `completed` / `failed` / `skipped` |
| `ai_error` | string | AI 识别失败时的错误信息，成功时为 `null` |
| `created_at` | datetime | 任务创建时间 |
| `local_completed_at` | datetime | 本地处理完成时间 |
| `face_completed_at` | datetime | 人物识别完成时间 |
| `ocr_completed_at` | datetime | 文字识别完成时间 |
| `ai_completed_at` | datetime | AI 识别完成时间 |

#### 3.7.5 进度事件通知

主进程通过 Electron IPC 向渲染进程推送进度更新事件：

| 事件名 | 数据 | 说明 |
|--------|------|------|
| `task:local-progress` | `{ md5, status, error? }` | 单张图片本地处理状态变化 |
| `task:face-progress` | `{ md5, status, error?, faceCount? }` | 单张图片人物识别状态变化，faceCount 为检测到的人脸数 |
| `task:ocr-progress` | `{ md5, status, error?, textLength? }` | 单张图片文字识别状态变化，textLength 为识别到的文字字符数 |
| `task:ai-progress` | `{ md5, status, error? }` | 单张图片 AI 识别状态变化 |
| `batch:summary` | `{ batchId, total, completed, failed, processing, pending }` | 批次总进度汇总（completed = 三线均完成的图片数） |

#### 3.7.6 并发控制

| 项目 | 说明 |
|------|------|
| **本地处理并发** | MD5 计算和元数据解析使用 Node.js Worker Threads 并行处理，默认并发数 = CPU 核心数 |
| **人物识别并发** | 使用独立 Worker Thread 运行 InsightFace ONNX 推理，默认并发 1（CPU 密集型，单张约 1-2 秒） |
| **文字识别并发** | 与人物识别共用 Worker Thread，串行执行（先人脸 → 再 OCR）。AI 方案时不占用本地队列 |
| **AI 识别并发** | 云端 API 默认并发 3（避免触发 API 限流），本地模型默认并发 1（受 GPU 显存限制） |
| **队列管理** | 使用优先级队列，新导入的图片排在队尾，手动重试的失败项排在队首 |
| **暂停/恢复** | 支持暂停 AI 识别和人物识别队列，本地处理（MD5 + 元数据）不可暂停（速度快，通常秒级完成） |

### 3.8 人物识别

> **优先级**: P1（重要功能）
>
> **技术方案**: InsightFace (ONNX Runtime) — 纯 CPU 运行，无需 GPU，工业级精度（LFW 99.82%）

基于本地人脸识别引擎，自动检测图片中的人脸并为同一人物分配持久 `person_id`，实现按人物浏览和筛选图片。

#### 3.8.1 功能概述

| 项目 | 说明 |
|------|------|
| **触发条件** | 图片导入后，作为本地处理流水线的一部分自动触发 |
| **运行模式** | 纯本地运行，使用 InsightFace ONNX 模型，无需联网，无需 GPU |
| **核心能力** | 人脸检测 → 特征提取 → 向量比对 → person_id 分配 |
| **多人支持** | 单张图片可检测多个人脸，每张人脸独立提取特征和匹配 |

#### 3.8.2 处理流程

```
图片导入（MD5 去重后）
    │
    ▼
人脸检测（InsightFace SCRFD 模型）
    ├── 检测到 N 张人脸 → 返回 N 个边界框坐标
    └── 未检测到人脸 → 标记 face_status = completed，faceCount = 0
    │
    ▼
对每张人脸提取 128 维特征向量（InsightFace ArcFace 模型）
    │
    ▼
与数据库已有特征向量逐一比对（欧氏距离）
    ├── 距离 < 阈值（默认 0.6）→ 匹配到已有 person_id
    └── 距离 ≥ 阈值 → 创建新 person_id
    │
    ▼
存储结果到 SQLite
    ├── persons 表：更新/新增人物记录
    ├── image_faces 表：记录每张人脸的归属关系
    └── tasks 表：更新 face_status = completed
    │
    ▼
用户可在「人物管理」面板中查看、命名、合并、拆分人物
```

#### 3.8.3 技术规格

| 项目 | 说明 |
|------|------|
| **人脸检测模型** | InsightFace SCRFD（Sample and Computation Redistribution for Face Detection） |
| **特征提取模型** | InsightFace ArcFace（推荐 `buffalo_l` 综合模型包） |
| **推理引擎** | ONNX Runtime（CPU 版），无需 CUDA/cuDNN |
| **模型格式** | ONNX，INT8 量化，模型体积 ~320MB |
| **特征向量** | 128 维浮点向量（ArcFace 输出） |
| **匹配阈值** | 欧氏距离 < 0.6 判定为同一人（可在设置中调整） |
| **运行环境** | 纯 CPU，无需 GPU；支持 Windows x64 |
| **集成方式** | 通过 Node.js Worker Thread 调用 ONNX Runtime，不阻塞主进程 |

#### 3.8.4 性能指标

| 环境 | 单张图片（单人） | 单张图片（3 人） | 说明 |
|------|:--------------:|:---------------:|------|
| Intel i5-8250U（4C8T） | ~1.5 秒 | ~3 秒 | 最低配置参考 |
| Intel i7-12700H（14C20T） | ~0.5 秒 | ~1 秒 | 推荐配置 |
| RTX 3060 GPU | <0.1 秒 | <0.3 秒 | GPU 加速（可选） |

> **首次导入性能**：1000 张图片（平均 1.5 人/张）首次建立人物索引约需 25-40 分钟（i5 CPU），后续增量导入秒级处理。

#### 3.8.5 数据模型

**persons 表（人物主表）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `person_id` | string (PK) | 人物唯一标识，自动生成的 UUID |
| `name` | string | 用户自定义的人物名称，默认为 `"人物 #N"` |
| `face_vector` | blob | 该人物的代表性特征向量（128 维 float32），取最近一次高质量人脸 |
| `face_count` | integer | 该人物出现的图片总数 |
| `first_seen_at` | datetime | 首次检测到该人物的时间 |
| `last_seen_at` | datetime | 最近一次检测到该人物的时间 |
| `thumbnail_md5` | string | 代表性头像图片的 MD5（取置信度最高的人脸） |

**image_faces 表（图片-人脸关联表）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | integer (PK) | 自增主键 |
| `md5` | string (FK → images) | 图片 MD5 |
| `person_id` | string (FK → persons) | 关联的人物 ID |
| `face_index` | integer | 该人脸在图片中的序号（从 0 开始） |
| `confidence` | float | 人脸检测置信度（0-1） |
| `bounding_box` | string | 人脸边界框 JSON：`{ "x", "y", "width", "height" }`（基于原图尺寸） |
| `face_vector` | blob | 该人脸的特征向量（用于后续合并/拆分时重新比对） |
| `created_at` | datetime | 记录创建时间 |

#### 3.8.7 边界框坐标

人物识别的人脸边界框坐标**直接基于原图尺寸**（不经过 AI 预处理压缩），因为人物识别使用原图进行推理，不存在压缩图映射问题。

#### 3.8.8 与 AI 语义识别的关系

| 维度 | 人物识别（3.8） | AI 语义识别（3.3） |
|------|:-------------:|:----------------:|
| **运行位置** | 本地 ONNX | 云端 API / 本地模型 |
| **处理内容** | 人脸检测 + 特征提取 | 物体/场景/活动/情绪 |
| **是否需要网络** | 否 | 云端模式需要 |
| **输入图片** | 原图 | 压缩图（≤500KB） |
| **并行运行** | ✅ 与 AI 识别完全独立，可并行 | ✅ 与人物识别完全独立 |
| **数据关联** | 通过 MD5 关联到同一张图片 | 通过 MD5 关联到同一张图片 |

> **设计决策**：人物识别不作为 AI 提示词的一个维度，而是作为独立的本地处理模块。原因：① InsightFace 的精度远高于通用 VL 模型的人脸能力 ② 本地运行无需网络、无 API 成本 ③ 特征向量可持久化，支持增量比对 ④ 用户可手动合并/拆分，需要确定性控制。

### 3.10 文字识别（OCR）

> **优先级**: P1（重要功能）
>
> **技术方案**: 可配置 — AI / PaddleOCR / Tesseract 三选一

识别图片中的文字内容，建立全文索引，支持用户通过图片中的文字检索图片。

#### 3.10.1 功能概述

| 项目 | 说明 |
|------|------|
| **触发条件** | 图片导入后自动触发，与人物识别共用本地处理队列串行执行 |
| **可配置方案** | 用户在设置中选择文字识别方案（AI / PaddleOCR / Tesseract） |
| **核心能力** | 文字检测 → 文字识别 → 全文索引 → 存储到数据库 |
| **全文索引** | 识别结果写入 SQLite FTS5 全文索引表，支持中文分词和模糊匹配 |
| **展示位置** | 仅在图片详情面板中展示，不在图片卡片上显示 |

#### 3.10.2 识别方案对比

| 方案 | 精度 | 速度 (CPU) | 安装体积 | 语言支持 | 网络要求 | 许可证 |
|------|:----:|:----------:|:-------:|:-------:|:-------:|:------:|
| **AI** | ⭐⭐⭐⭐ | ~3-5s/张 | 0（复用 AI 服务） | 90+ | 需要（云端模式） | 取决于 AI 服务 |
| **PaddleOCR** | ⭐⭐⭐⭐⭐ | ~5s/张 | ~500MB | 100+ | 不需要 | Apache 2.0 |
| **Tesseract** | ⭐⭐½ | ~0.16s/张 | ~10MB | 100+ | 不需要 | Apache 2.0 |

> **默认方案**: PaddleOCR（精度最高、中文支持最强、纯本地运行）

#### 3.10.3 统一输出格式

无论选择哪种方案，输出格式统一：

```json
{
  "text": {
    "content": "图片中识别到的完整文字内容（多区域按阅读顺序拼接，区域间用换行符分隔）",
    "regions": [
      {
        "text": "区域文字内容",
        "confidence": 0.92,
        "boundingBox": { "x": 50, "y": 20, "width": 300, "height": 40 }
      }
    ]
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `text.content` | string | 完整文字内容，多区域按从上到下、从左到右顺序拼接，区域间用 `\n` 分隔 |
| `text.regions[].text` | string | 单个文字区域的内容 |
| `text.regions[].confidence` | number | 置信度，范围 0–1 |
| `text.regions[].boundingBox` | object | 文字区域边界框（基于原图尺寸的像素值） |

> **AI 方案特殊处理**：当选择 AI 方案时，文字识别结果由 AI 模型在四维度识别中一并返回（需在 prompt 中保留文字识别规则），`text` 字段直接附加到 AI 返回结果中。本地 OCR 方案的结果由 OCR 模块独立写入数据库。

#### 3.10.4 处理流程

```
图片导入（MD5 去重后）
    │
    ▼
进入本地处理队列（与人物识别共用）
    │
    ├──▶ 人物识别（InsightFace）
    │       完成后 ↓
    │
    └──▶ 文字识别（根据配置方案）
            │
            ├── AI 方案：文字识别结果已包含在 AI 返回中，无需额外处理
            │
            ├── PaddleOCR 方案：
            │       文字检测（PP-OCR 检测模型）
            │           → 文字识别（PP-OCR 识别模型）
            │           → 格式化为统一输出格式
            │
            └── Tesseract 方案：
                    文字检测 + 识别（Tesseract 引擎）
                        → 格式化为统一输出格式
            │
            ▼
    写入 text 字段到 AI 分析结果表
    写入全文索引到 FTS5 表
    如果 text 非 null → detectedTypes 追加 "text"
    │
    ▼
    标记 ocr_status = completed
```

#### 3.10.5 与人物识别的队列关系

| 项目 | 说明 |
|------|------|
| **共用队列** | 人物识别和文字识别（本地 OCR）共用同一个本地处理队列 |
| **串行执行** | 同一张图片先执行人物识别，完成后再执行文字识别 |
| **原因** | 两者都是 CPU 密集型任务，共用一个 Worker Thread 可减少资源占用和内存开销 |
| **AI 方案例外** | 当文字识别选择 AI 方案时，OCR 结果由 AI 识别队列返回，不占用本地队列 |
| **进度展示** | 在进度面板中，人物识别和文字识别各自有独立状态图标 |

#### 3.10.6 设置项

| 设置项 | 选项 | 默认值 | 说明 |
|--------|------|--------|------|
| **文字识别方案** | AI / PaddleOCR / Tesseract | PaddleOCR | 切换后仅影响新导入的图片 |
| **OCR 语言** | 自动检测 / 手动选择 | 自动检测 | PaddleOCR 和 Tesseract 使用 |
| **最小置信度** | 0-1 滑块 | 0.5 | 低于此置信度的文字区域不存储 |

#### 3.10.7 全文索引

| 项目 | 说明 |
|------|------|
| **索引引擎** | SQLite FTS5（Full-Text Search 5） |
| **索引内容** | `text.content`（完整文字内容） |
| **分词支持** | 内置 Unicode 分词器，支持中文、英文、日文等 |
| **搜索方式** | MATCH 查询，支持模糊匹配和短语搜索 |
| **搜索入口** | 主界面搜索框，输入关键词后同时搜索文件名、标签、分类和图片内文字 |
| **结果高亮** | 搜索结果中高亮匹配的文字片段 |

| MD5 | 消息摘要算法第5版，用于文件唯一标识，始终基于原始图片文件计算 |
| 图片预处理压缩 | AI 识别前对原图进行缩放和编码压缩，将几 MB 原图压缩到 ≤500KB，避免 AI 调用失败 |
| 压缩图 | 经预处理压缩后的图片，仅用于 AI 识别输入，不替代原图存储 |
| 边界框坐标映射 | 将 AI 基于压缩图返回的边界框坐标按比例换算回原图尺寸 |
| 人物识别 | 基于 InsightFace ONNX 的本地人脸检测与特征提取，为同一人物分配持久 person_id |
| person_id | 人物唯一标识，由系统自动分配或用户合并/拆分后更新 |
| 特征向量 | 128 维浮点向量，用于表示人脸的生物特征，通过欧氏距离判断是否同一人 |
| 匹配阈值 | 欧氏距离阈值（默认 0.6），低于此值判定为同一人物 |
| SQLite | 轻量级嵌入式关系型数据库 |
| better-sqlite3 | SQLite 的 Node.js 同步绑定库 |
| 沙箱 | 隔离执行环境，限制代码的访问权限 |
| 置信度 | AI 模型对识别结果的确定程度，范围 0–1 |
| System Prompt | 系统提示词，定义 AI 模型的角色、分析规则和输出约束 |
| User Prompt | 用户提示词，携带具体图片信息和 JSON 模板，由应用动态填充 |
| 后处理校验 | 对模型输出进行格式修正、边界检查、空值规范化的代码层处理 |
| 钳制（Clamp） | 将数值限制在指定范围内的操作，如置信度限制在 [0, 1] |
